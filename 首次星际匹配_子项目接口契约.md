## 3. 阶段一后端接口契约（已实现）

### 3.1 `POST /v1/game/match`

**文件**: `backend/src/api/routes/game_routes.py` → `fleet_match`

**前置**:
- 已登录 JWT
- 已完成人格测试（`personality_type` 非空）
- 已在舰队中，且舰队内 **≥1 名其他成员**

**行为（v0）**:
- 对舰队内其他成员按人格互补规则打分（0–100）
- 返回最高分候选人；**不**自动完成 mission；**不**持久化

**请求**: `{}`（无 body）

**响应（v0）**:
```json
{
  "matched": true,
  "match": {
    "user_id": "uuid",
    "name": "显示名",
    "personality_type": "星云艺术家",
    "personality_emoji": "🎨",
    "match_score": 95,
    "reason": "人格互补 · 创造力遇见深度逻辑"
  },
  "self": {
    "user_id": "uuid",
    "personality_type": "黑洞程序员",
    "personality_emoji": "🖤"
  },
  "fleet_id": "uuid",
  "fleet_name": "星际舰队",
  "candidates_considered": 2
}
```

**错误码**:

| `error` | 条件 |
|---------|------|
| `personality_required` | 未做人格测试 |
| `fleet_required` | 未加入舰队 |
| `fleet_too_small` | 舰队仅自己 |

**评分档位（v0 规则，阶段二复用）**:

| 分数 | 场景 |
|------|------|
| 95 | 互补人格对（如星云艺术家 ↔ 黑洞程序员） |
| 72 | 同型人格 |
| 58 | 异型非互补 |
| 0–40 | 对方未测人格等 |

### 3.2 `POST /v1/game/mission-complete`

```json
// 请求
{ "mission_id": "match", "xp_reward": 40 }

// 响应
```

`mission_completions` 表 `UNIQUE(user_id, mission_id)` — 重复调用返回 `409`。

## 5. 阶段二产品目标：共识裂变模型

### 5.1 商业意图

匹配的核心目的不是「凑够三人舰队就分配队友」，而是：

1. **共识验证** — 成员间须达到高度契合标准，才算「验证组队成功」
2. **裂变增长** — 未达标时鼓励成员主动传播（一传十，十传百），扩大候选池
3. **循环直至成功** — 传播 → 新成员入舰队/入图 → 再扫描 → 直至共识达成

### 5.2 核心原则

| # | 原则 |
|---|------|
| P1 | **匹配 = 验证**，不是自动分配 |
| P2 | **失败 = 增长机会**，不是死胡同 |
| P3 | **成功 = 集体成就**，强调「我们达成了共识」 |
| P4 | **share 是 match 的出口**，不是平行无关任务 |

### 5.3 阶段二用户流程

```
完成人格测试
    ↓
加入/创建舰队（验证池，人数够 ≠ 自动成功）
    ↓
发起「共识验证扫描」 POST /v1/game/match
    ↓
┌─ 契合度 ≥ CONSENSUS_THRESHOLD (默认 85)?
│   ├─ 是 → 进入「双向共识确认」流程
│   │         ├─ 对方在窗口期内认可 → consensus_verified
│   │         │     → 允许 mission-complete('match') → 成就「验证组队成功」
│   │         └─ 对方未认可 → 等待 / 提醒 / 重新扫描
│   └─ 否 → 「信号未达共振标准」
│             → 主 CTA：发送星际信号（舰队邀请 + 人格分享）
│             → 记录传播事件，扩大扫描范围
│             → 引导继续拉人 → 回到扫描
└─ 循环直至 consensus_verified
```

### 5.4 共识门槛（硬标准）

| 契合度 | 状态码 | 用户文案 | 系统行为 |
|--------|--------|---------|---------|
| ≥ 85 | `consensus_passed` | 高度契合 · 可进入双向确认 | 展示确认页；**禁止**直接 mission-complete |
| 60–84 | `consensus_weak` | 信号接近，尚未共振 | 不推荐完成；引导分享 + 再扫描 |
| < 60 | `consensus_failed` | 星图稀疏，需扩张传播 | 强引导分享；可选奖励「传播 XP」 |

> 阈值 `CONSENSUS_THRESHOLD` 建议默认 **85**，可通过环境变量或配置下发，便于内测调参。

### 5.5 双向共识（软标准，防刷）

| 步骤 | 说明 |
|------|------|
| 1 | 用户 A 扫描得到候选人 B，且 `match_score ≥ 85` |
| 2 | 系统创建一条 `pending` 共识记录（A→B，含分数、舰队、过期时间） |
| 3 | B 收到通知（或进入 match 页看到「待你认可的共振请求」） |
| 4 | B 调用 `POST /v1/game/match/acknowledge` 认可该配对 |
| 5 | （可选强化）B 对 A 的反向扫描也 ≥ 85 |
| 6 | 双方认可后状态变为 `verified` → A、B 均可（或仅发起方）完成 `match` mission |

**防刷**: 未双向确认前，前端不得调 `mission-complete`；后端阶段二应对 `mission-complete(match)` 校验是否存在 `verified` 共识记录。

### 5.6 裂变引导（未达标出口）

未达标页必须提供：

| 元素 | 实现复用 |
|------|---------|
| 舰队邀请码复制 | `fleet.invite_code` / `GET /v1/game/profile` |
| 人格分享文案 | `shared-core/utils/share.ts` → `getShareText()` |
| 分享任务入口 | Hub `share` 任务 / `referral/create` + `profile_share` |
| 传播进度 | 「已邀请 N 人完成人格测试 · 再邀请 M 人解锁广域扫描」 |

**传播计数**（阶段二新增）:
- 事件：`invite_accepted`、`profile_share_view`、`referral_used`（已有 analytics 可复用）
- 成对指标：`spread_count` — 因当前用户传播而新完成人格测试的人数
- 产品规则（建议）：每 +1 有效传播，`match` 扫描候选池扩展提示升级（文案层）；阶段二可不改算法，先改 UI 与任务引导

### 5.7 成就与 XP（阶段二）

| 事件 | XP 建议 | 成就文案 |
|------|---------|---------|
| 共识验证通过（双向确认后） | +40（`match` mission） | 🎯 验证组队成功！共识共振已达成 |
| 高度契合（≥95）且首次 verified | +10 bonus（可选新 mission 或 analytics 事件） | ✨ 星际完美共振 |
| 有效传播 +1（未达标场景） | +5 `spread_bonus`（可选，不累加进 match mission） | 📡 星图扩张中 |

### 5.8 与 `team` / `share` 任务的关系

| 任务 | 阶段二定位 |
|------|-----------|
| `team` | 组建验证池（≥3 人完成 team mission 仍可作为 Hub 解锁条件） |
| `match` | **仅**在 `consensus_verified` 后可完成 |
| `share` | 未达标 match 的**主 CTA**；可与传播 bonus 联动，而非孤立 +30 XP |

## 6. 阶段二接口契约（拟定，待开发）

### 6.1 `POST /v1/game/match` 响应扩展

在 v0 响应基础上**新增字段**（向后兼容）：

```json
{
  "matched": true,
  "match": { "...v0 同结构..." },
  "self": { "...v0 同结构..." },
  "fleet_id": "...",
  "fleet_name": "...",
  "candidates_considered": 2,

  "consensus_threshold": 85,
  "consensus_status": "consensus_passed",
  "consensus_passed": true,
  "can_complete_mission": false,
  "pending_consensus_id": "uuid-or-null",
  "spread_hint": {
    "show_share_cta": false,
    "message": null,
    "spread_count": 3,
    "spread_target": 5
  }
}
```

**`consensus_status` 枚举**:

| 值 | 含义 |
|----|------|
| `consensus_passed` | 分数 ≥ 阈值，待双向确认 |
| `consensus_weak` | 60–84 |
| `consensus_failed` | < 60 |
| `consensus_verified` | 双向已确认，可完成 mission |

**`can_complete_mission`**: 仅当 `consensus_verified` 时为 `true`。

### 6.2 新增 `POST /v1/game/match/acknowledge`

**用途**: 候选方认可发起方的共识验证请求。

**请求**:
```json
{
  "consensus_id": "uuid",
  "action": "accept"
}
```

**响应**:
```json
{
  "consensus_id": "uuid",
  "status": "verified",
  "match_score": 95,
  "initiator_id": "uuid-a",
  "candidate_id": "uuid-b",
  "verified_at": "2026-07-12T12:00:00"
}
```

### 6.3 新增 `GET /v1/game/match/consensus`

**用途**: 查询当前用户待处理 / 已完成的共识记录。

**响应**:
```json
{
  "pending_incoming": [],
  "pending_outgoing": [],
  "verified": []
}
```

### 6.4 `POST /v1/game/mission-complete` 门控（阶段二）

对 `mission_id === "match"` 增加服务端校验：

```
若无 status=verified 且未过期的 consensus 记录 → 403 consensus_required
```

### 6.5 新增数据库表（拟定）

```sql
-- 共识验证记录
CREATE TABLE IF NOT EXISTS match_consensus (
    id              TEXT PRIMARY KEY,
    fleet_id        TEXT NOT NULL,
    initiator_id    TEXT NOT NULL,
    candidate_id    TEXT NOT NULL,
    match_score     INTEGER NOT NULL,
    status          TEXT DEFAULT 'pending',  -- pending | verified | expired | rejected
    reason          TEXT,
    created_at      TEXT DEFAULT (datetime('now')),
    verified_at     TEXT,
    expires_at      TEXT NOT NULL,
    FOREIGN KEY (initiator_id) REFERENCES users(id),
    FOREIGN KEY (candidate_id) REFERENCES users(id)
);

CREATE INDEX IF NOT EXISTS idx_match_consensus_initiator ON match_consensus(initiator_id, status);
CREATE INDEX IF NOT EXISTS idx_match_consensus_candidate ON match_consensus(candidate_id, status);

-- 传播计数（可选，亦可先复用 product_events）
CREATE TABLE IF NOT EXISTS match_spread_events (
    id              TEXT PRIMARY KEY,
    user_id         TEXT NOT NULL,
    event_type      TEXT NOT NULL,   -- invite_sent | invite_joined | profile_shared
    related_user_id TEXT,
    created_at      TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

## 9. 总结

| 层面 | 阶段一 v0 | 阶段二目标 |
|------|-----------|-----------|
| 产品定位 | 舰队内算分闯关 | **共识验证 + 裂变增长** |
| 匹配 API | ✅ `POST /v1/game/match` | 扩展共识字段 + acknowledge |
| 任务完成 | 用户点确认即完成 | **verified 后才能 mission-complete** |
| 未达标 | 无特殊处理 | **引导 share，扩大传播** |
| 前端 | ✅ 双端匹配页 | 三分流 UI + 传播页 |
| 数据 | 无持久化 | `match_consensus` 表 |
| 测试 | 5 个算法用例 | + 状态机 + 门控用例 |

**阶段一**解决了「能跑通」；**阶段二**解决「匹配服务于裂变推广」的产品目标。开发请优先按 §7 Diff 清单 P0 项对齐，再迭代 P1 传播体验。
