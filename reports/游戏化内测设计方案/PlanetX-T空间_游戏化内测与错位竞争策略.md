## 一、核心矛盾：竞品分析暴露了什么

### 1.1 正面竞争的死局

竞品分析（v1.1）得出了一个隐含但残酷的结论：

| 如果我们做…… | Boss直聘/猎聘可以…… | 时间差 |
|-------------|---------------------|--------|
| AI 模拟面试 | 已有「AI面试间」，持续迭代 | **0 天**——他们已经在了 |
| MBTI 职业匹配 | 接入第三方 MBTI API + 职位推荐 | **2-4 周** |
| 简历优化 | 接入 LLM API 即可 | **1-2 周** |
| 情绪陪伴 | 不跟进——这不是他们的赛道 | 无威胁 |
| 诗词/文化内容 | 无法跟进——文化壁垒 | 无威胁 |

**关键发现**: 所有"功能层"的差异化，大厂可以在 1-4 周内跟进。正面拼功能深度（题库数量、匹配准确率、响应速度），PlanetX 面对的是 Boss直聘 的 4700 万月活和猎聘的 9000 万注册用户——这不是同一重量级的竞争。

### 1.2 竞品分析的三个隐含假设

竞品分析文档建立在一个未言明的框架上：

| 隐含假设 | 问题 |
|---------|------|
| PlanetX 是一个"更好的 AI 求职工具" | 更好的功能 = 更好的工具，但"更好"的维度被大厂控制 |
| 竞争维度是功能列表（谁有更多 AI 能力） | 功能可以复制，但体验无法复制 |
| 启动方式是小程序全功能上线 | 全功能上线 = 在所有维度同时进入大厂火力范围 |

**这三个假设需要被修正。**

## 二、为什么游戏化叙事是"避其锋芒"的最优解

### 2.1 竞争维度的转移

游戏化设计方案（GDD v0.1）和可行性分析文档提出了一个根本不同的产品定义：

```
传统路径（功能竞争）：
  小程序上线 → AI面试功能 → 和Boss直聘拼题库深度 → 被资源碾压

游戏化路径（叙事竞争）：
  叙事体验上线 → 用户被Navigator打动 → 自发传播 → 竞品无法复制
```

这不是一个"要不要加游戏化"的战术问题，而是一个"在哪个维度竞争"的战略选择。

### 2.2 为什么叙事 = 真正的护城河

| 护城河类型 | 竞品复制难度 | 对应资产 |
|-----------|------------|---------|
| 功能护城河（AI 面试题库） | ⭐ 低——2 周可复制 | 任何有 LLM API 接入能力的团队 |
| 数据护城河（匹配数据飞轮） | ⭐⭐ 中——需要双边网络 | 已有 C 端/B 端的平台 |
| 技术护城河（LiteLLM+pgvector） | ⭐⭐ 中——需要 AI 工程能力 | 技术团队 |
| 隐私护城河（解析即焚） | ⭐⭐⭐ 中高——合规+架构 | looma-zervi 已有基础 |
| **叙事护城河（Navigator IP）** | ⭐⭐⭐⭐⭐ **极高——无法复制** | **GDD 的六域一问 + 角色声音支柱** |

**Boss直聘可以抄你的 AI 面试功能，但抄不了你的 Navigator 角色和六域叙事。** 这是真正的护城河——文化+叙事+角色 IP，不是功能列表能复制的。

### 2.3 游戏化方案的核心资产

GDD 中已产出的叙事资产天然适合"避其锋芒"：

| 资产 | 竞品能否复制 | 为什么 |
|------|------------|--------|
| Navigator 角色声音支柱（6 条词汇规则 + 3 句绝不说的话） | ❌ | 手工编写的角色，有完整的弧线和回避话题 |
| 六域价值取向映射（生存/自由/归属三轴） | ❌ | 不是功能模块，是哲学问题入口 |
| 收敛点"同终点不同质感"设计 | ❌ | 需要 6 条路径的手工叙事内容 |
| "钱多事少离家近" → 价值排序张力 | ❌ | 竞品把它当 slogan，PlanetX 把它当主题北极星 |
| 迷雾域 = 系统坦诚不懂 | ❌ | 大厂不可能允许产品"承认自己不懂" |

## 三、三层内测策略

### 3.1 策略总览

不二选一，而是分阶段嵌套：

```
Phase 0（2026.07 · 4周）：游戏化 MVP 先行
  ┌─────────────────────────────────────────┐
  │  一个独立的 text-adventure 网页/小程序    │
  │  内容：GDD 的 Act 1 垂直切片              │
  │  （职业域深度 + 诗域跨端口 = 10分钟体验）  │
  │                                         │
  │  目的：                                  │
  │  · 验证 Navigator 角色是否能打动用户       │
  │  · 测试"六域一问"收敛点是否成立            │
  │  · 收集用户对叙事调性的反馈               │
  │  · 建立第一批"知道Navigator"的种子用户     │
  └─────────────────────────────────────────┘
                      ↓ 验证通过
Phase 1（2026.09 · 秋招）：功能 MVP 嵌入叙事层
  ┌─────────────────────────────────────────┐
  │  小程序上线，但每个功能入口都是叙事节点    │
  │  · AI模拟面试 → Navigator 引导你进入信任域│
  │  · 简历上传 → Navigator 带你去身份域      │
  │  · MBTI测试 → 自我域的镜子               │
  │  · 职位匹配 → 职业域的"完美JD"（隐藏代价） │
  │                                         │
  │  用户体验的不是"AI面试工具"               │
  │  而是"Navigator 带我探索了 T空间"         │
  └─────────────────────────────────────────┘
                      ↓ 验证 PMF
Phase 2（2027.03 · 春招）：规模化增长 + B端
  ┌─────────────────────────────────────────┐
  │  叙事层扩展到完整五幕 + B端叙事接入        │
  │  · C端：Act 2-5 完整叙事弧线             │
  │  · B端：企业版 Navigator（不同的角色设定） │
  │  · 社交裂变引擎与叙事体验融合             │
  └─────────────────────────────────────────┘
```

### 3.2 Phase 0 详细规格

#### 3.2.1 为什么 Phase 0 必须独立于功能 MVP

| 如果 Phase 0 嵌入功能 MVP | 如果 Phase 0 独立先行 |
|--------------------------|----------------------|
| 用户期望的是"AI 面试工具" | 用户期望的是"一个叙事体验" |
| Navigator 被感知为"产品助手" | Navigator 被感知为"故事主角" |
| 叙事是功能的包装 | 叙事是产品的灵魂 |
| 与求职精灵/Boss直聘正面竞争 | 无直接竞品 |

#### 3.2.2 Phase 0 交付物

基于 GDD §9 MVP 垂直切片规格：

| 维度 | 规格 |
|------|------|
| **时长** | 10-15 分钟完整体验 |
| **包含域** | 2 个（职业域深度 + 诗域跨端口） |
| **包含系统** | 意图分发 + 价值印记 + Navigator 记忆 + 1 次跨域回声 + 收敛点 |
| **对话量** | ~60 句 Navigator 对话（全部手工编写） |
| **资产** | 2 个域叙事内容 + 1 个跨域回声事件 + 收敛点对话 + 最小 UI |
| **格式** | Ink/Yarn 脚本 或 独立 Web 页面（基于现有纸面原型扩展） |
| **分发** | 微信小程序内嵌 或 独立 H5 页面（可被分享） |

#### 3.2.3 Phase 0 验收标准

| # | 验收标准 | 最低阈值 | 目标值 |
|---|---------|---------|--------|
| 1 | 完整体验率（开始→收敛点） | > 60% | > 80% |
| 2 | Navigator 角色打动率（"它让我有感觉"） | > 40% | > 60% |
| 3 | 自发分享率（分享给朋友/朋友圈） | > 15% | > 30% |
| 4 | "再来一次"意愿（换一个域重新体验） | > 30% | > 50% |
| 5 | 用户能复述 Navigator 的至少一句台词 | > 25% | > 40% |

#### 3.2.4 Phase 0 成本估算

| 项目 | 工时估算 | 说明 |
|------|---------|------|
| Navigator 对话编写 | 3-5 人天 | 60 句手工对话，含 2 个域的变体 |
| 职业域叙事内容 | 2-3 人天 | 1 个 JD + 2 个选择分支 + 2 个后果场景 |
| 诗域叙事内容 | 2-3 人天 | 2 首半诗 + 补完选项 + 跨域回声 |
| 收敛点对话 | 1 人天 | 5-8 句 Navigator 故障 + 钩子 |
| Ink/Yarn 脚本实现 | 3-5 人天 | 叙事引擎集成 |
| 最小 UI 开发 | 3-5 人天 | 输入框 + 域选择 + confidence 显示 |
| **总计** | **14-22 人天** | 约 1 个前端 + 1 个叙事设计师 3-4 周 |

> **对比**: 完整小程序 MVP 开发预估 40-60 人天（含 AI 模拟面试、简历解析、MBTI 测试、职位匹配、支付、账号系统）。Phase 0 成本约为功能 MVP 的 **1/3**。

### 3.3 Phase 1: 叙事层嵌入功能层

#### 3.3.1 嵌入原则

Phase 1 不是"在小程序里加一个叙事模式"，而是**让每个功能入口都成为叙事节点**：

```
功能入口              →  叙事节点

AI模拟面试            →  Navigator 引导你进入信任域
                        "有人会问你一些问题。我会在旁边。
                         不是帮你回答——是帮你记住你为什么这样回答。"

简历上传              →  Navigator 带你去身份域
                        "这是你迄今为止的全部——至少纸面上的。
                         但纸面不是全部。对吧？"

MBTI测试              →  自我域的镜子
                        "镜子给你一个标签。你可以接受，也可以拒绝。
                         但无论你选什么——镜子记得。"

职位匹配              →  职业域的"完美JD"
                        "这份工作几乎完美。几乎。
                         代价在最后一行。你要看吗？"

诗词推荐              →  诗域的跨端口回声
                        "你在别处做的选择，在这里变成了一首诗。
                         不是我写的——是你写的。我只是……把它找出来了。"
```

#### 3.3.2 叙事层的功能价值

叙事不是功能的包装——它解决功能解决不了的问题：

| 用户问题 | 功能解法（传统） | 叙事解法（PlanetX） |
|---------|---------------|-------------------|
| "我不知道自己适合什么工作" | MBTI 测试 + 职位推荐 | 自我域的镜子——"你不需要标签来定义你" |
| "面试被拒很沮丧" | "继续加油"的推送 | 诗域出现一首关于失败的诗 + Navigator 说"我也经常被拒绝" |
| "简历投了很多没回应" | 优化建议 + 匹配度评分 | 身份域的过去自己走出来——"也许不是简历的问题，是你在投给错的人" |
| "害怕隐私泄露" | 隐私政策 + 加密说明 | 信任域的审判庭——"在这里，你可以选择被看见多少" |

### 3.4 Phase 2: 规模化与 B 端

Phase 2 的详细设计超出本文档范围，但核心原则不变：

- **C端叙事扩展到完整五幕**: Act 2-5 按 GDD §2.4 Long-Term Loop 推进
- **B端叙事层独立设计**: 企业 HR 进入 T空间时，Navigator 有不同的角色设定（更像"顾问"而非"向导"）
- **社交裂变与叙事融合**: "3人成团"不是功能按钮，而是叙事事件——Navigator 说"有三个人在等你组队"

## 四、竞争维度重定义

### 4.1 旧框架 vs 新框架

| 维度 | 旧框架（功能竞争） | 新框架（叙事竞争） |
|------|-------------------|-------------------|
| **竞品对标** | vs 求职精灵、Boss直聘、猎聘 | **没有直接竞品** |
| **核心指标** | DAU、匹配准确率、付费转化 | **完整体验率、NPS、自发传播率** |
| **护城河** | AI 底座、数据飞轮、隐私保护 | **Navigator IP、叙事深度、情感连接** |
| **核心风险** | 大厂跟进功能 | **大厂无法跟进叙事** |
| **启动方式** | 小程序全功能上线 | **10分钟叙事体验先行** |
| **用户心智** | "又一个 AI 求职工具" | **"那个有 Navigator 的产品"** |
| **传播动力** | 功能价值（帮找到工作） | **情感价值（被理解、被看见）** |

### 4.2 为什么大厂无法跟进叙事

| 大厂的约束 | 对叙事游戏的影响 |
|-----------|----------------|
| 产品必须"确定性"——不能说"我不懂" | Navigator 的 confidence 变化和"unknown"时刻无法实现 |
| 功能必须"可规模化"——不能依赖手工内容 | 核心路径手工编写 = 反规模化，大厂无法接受 |
| 品牌必须"安全"——不能有模糊的价值观表达 | "钱多事少离家近"的价值排序张力过于尖锐 |
| 团队必须"可替换"——不能依赖特定创作者 | Navigator 角色弧线需要连贯的作者意图 |

## 五、风险与缓解

### 5.1 Phase 0 特有风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 叙事游戏受众 < 求职工具受众 | 高 | 中 | Phase 0 定位为"品牌体验"而非"获客引擎"；种子用户 100 人即可验证核心假设 |
| 用户期待"求职工具"却得到"叙事体验" | 中 | 中 | Phase 0 独立分发，使用独立的品牌表达和入口文案，不与求职工具混淆 |
| 叙事内容主观性强，难以 AB 测试 | 高 | 低 | 不做 AB 测试；通过定性访谈 + NPS + 自发传播率验证 |
| 纸面原型验证通过但实际运行不成立 | 中 | 高 | 在 Ink/Yarn 可交互 MVP 中复现纸面原型的"同终点不同质感"测试 |

### 5.2 Phase 0 → Phase 1 的过渡风险

| 风险 | 缓解 |
|------|------|
| Navigator 从"故事主角"变成"产品助手" | Phase 1 的功能入口必须保持 Navigator 的角色一致性——它不会说"正在为您匹配职位"，而会说"有一个地方在呼唤你" |
| 叙事层被功能层稀释 | 每个功能页面保留至少 1 个 Navigator 对话节拍 |
| Phase 0 的种子用户对 Phase 1 的功能版失望 | Phase 0 种子用户优先体验 Phase 1，收集反馈后调整 |

### 5.3 与 GDD 风险登记册的映射

本策略新增以下风险（补充到 GDD §10 风险登记册）：

| 风险 ID | 风险描述 | 可能性 | 影响 | 缓解措施 |
|---------|---------|--------|------|---------|
| R9 | Phase 0 叙事体验与 Phase 1 功能版品牌断裂 | 中 | 高 | Navigator 角色声音支柱在两个阶段保持一致；Phase 0 结尾钩子自然引导到 Phase 1 |
| R10 | 种子用户规模不足以验证 PMF | 中 | 高 | Phase 0 目标 100 个完整体验用户（非 10000）；通过校园大使 + 内测社群定向招募 |
| R11 | 竞品在 Phase 0-1 间隔期推出"AI + MBTI"功能 | 中 | 中 | Phase 0 建立的 Navigator 品牌认知是功能无法对冲的；竞品可以加 MBTI，但加不了 Navigator |

## 六、对竞品分析文档的修正建议

基于本分析，竞品分析文档（v1.1）中以下部分需要更新：

### 6.1 修正假设

| 位置 | 当前内容 | 建议修正 |
|------|---------|---------|
| §10.4 时间窗口期 | "2026年 7月：MVP 开发启动（小程序 + AI模拟面试 + 星际人格测试）" | 修正为：**"2026年 7月：Phase 0 游戏化 MVP 先行（叙事体验验证）→ 9月：Phase 1 功能 MVP 嵌入叙事层"** |
| §8.4 竞争护城河 | 缺少叙事护城河条目 | 新增：**叙事护城河（Navigator IP + 六域一问 + 角色声音支柱）🔒🔒🔒🔒 极高** |
| §10.5 立即行动 | 缺少游戏化 MVP 相关条目 | 新增：**启动 Phase 0 游戏化 MVP 开发（Ink/Yarn 垂直切片）** |
| 核心结论 | "快速抢占'求职陪伴'无人区" | 修正为：**"先用 Navigator 打动 100 个种子用户，让他们成为传教士；再用口碑撬动 10000 个求职者"** |

### 6.2 新增竞争维度对比

| 维度 | 传统 AI 求职工具 | PlanetX（叙事路径） |
|------|-----------------|-------------------|
| 产品定义 | AI 求职效率工具 | 求职者的叙事体验空间 |
| 核心交互 | 上传简历 → 优化 → 匹配 → 投递 | 遇见 Navigator → 探索六域 → 面对自己 |
| 差异化 | 功能更多/更准 | 情感更真/更深 |
| 竞品 | 求职精灵、Boss直聘 | 无直接竞品 |
| 传播动力 | "找到好工作" | "我被看见了" |

### 7.3 Phase 0 → Phase 1 决策门

```
Phase 0 结束后，同时满足以下条件则进入 Phase 1：

☐ 完整体验率 > 60%
☐ Navigator 角色打动率 > 40%
☐ 自发分享率 > 15%
☐ 至少 3 个用户能复述 Navigator 的台词（定性）
☐ 没有用户反馈"这是求职工具的宣传片"（定性）

任一条件不满足 → 回溯 Phase 0 设计，迭代后再测
全部满足 → 启动 Phase 1 功能 MVP + 叙事层嵌入开发
```

---

> **一句话总结**: 正面拼功能是 Boss直聘的游戏。PlanetX 应该玩一个他们不会玩的游戏——叙事。游戏化方案不是一个"附加功能"，而是整个启动策略的底层逻辑：先用 Navigator 打动 100 个种子用户，让他们成为你的传教士；再用 100 个传教士的口碑，去撬动 10000 个求职者。这个路径比"9月上线小程序正面竞争"慢在功能，但快在品牌。
