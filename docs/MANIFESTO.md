## 一、我们为什么存在

### 1.1 我们看到的根本问题

今天的职业信任体系，是一场**信息不对称的结构性不公**：

| 有的人 | 有的人 |
|--------|--------|
| 名校毕业、大厂履历 | 三线城市、自学成才 |
| 圈内人脉、猎头推荐 | 没有介绍人、没有背书 |
| 简历本身就是信任凭证 | 能力很强，但无法"证明"自己 |

这两类人在能力上可能没有本质差距，但在现有体系中，前者天然拥有信任，后者几乎没有任何途径向雇主"证明"自己的可信度。

**这不是一个招聘效率问题，这是一个信任分配问题。**

现有的职业平台——LinkedIn、脉脉、Boss 直聘——解决的从来都是"如何更快地找到人"或"如何把简历写得更漂亮"，而不是"如何让一个人的真实行为成为可验证的信任凭证"。它们优化的是匹配效率，我们追问的是信任正义。

### 1.2 我们的回答

> **让信任的获取不再依赖出身，而是依赖行为。**

我们不是在做一个更好用的招聘工具。我们在建立一套**用真实行为记忆来验证职业信任的基础设施**。名校和大厂不再垄断信任的定义权，行为本身成为通行证。

### 1.3 社会价值的三层表达

| 层 | 对谁说 | 我们提供的价值 |
|----|--------|----------------|
| **个体层** | 求职者，尤其是"没有信号"的人 | 把行为轨迹变成可验证的信任资产，让能力被看见 |
| **组织层** | HR、合伙人、企业 | 不再依赖关键词过滤和直觉判断，获得行为验证过的候选人画像 |
| **社会层** | 职业生态 | 让信任的分配从"出身导向"转向"行为导向"——这是职业公平的基础设施 |

### 1.4 我们刻意不做的

| 常见做法 | 我们的立场 | 原因 |
|----------|------------|------|
| 好友数 = 信任度 | **不做** | 结构可达 ≠ 信任，这不是价值判断，是逻辑判断 |
| 简历 AI 润色 | **不做** | 美化 PDF 不等于验证能力 |
| 会员等级 = 信用分 | **不做** | 身份门槛 ≠ 行为验证，付费不能购买信任 |
| 社交图谱扩张 KPI | **不做** | 图是关系上下文，不是信任本体 |
| 竞品功能堆砌 | **不做** | 体验可参考，信任本体不同——我们不模仿 |

### 1.5 我们的定位：信任档案平台

**我们不是数据提供商，不是招聘平台——我们是信任档案平台。**

| 定位 | 核心资产 | 输出形态 | 我们在哪里 |
|------|----------|----------|------------|
| 数据提供商 | 原始行为数据 | API / 报告 | ❌ 数据在库中但没有被"产品化"呈现 |
| 招聘平台 | 简历 + JD 匹配 | 匹配列表 + improve 报告 | ⚠️ 这是我们的数据入口，不是产品出口 |
| **信任档案平台** | 行为记忆体 + 可验证声明 | 个人信任档案页、候选人行为画像、验证 API | 🎯 **这才是我们的产品形态** |

招聘是我们的第一个应用场景——但不是产品形态。简历上传和职位匹配是**数据源头**，Trust Agent 生成的声明卡是**信任中间件**，个人信任档案页和候选人多维画像是**产品出口**。

**生态服务的"输出"不是报告，是被看见的信任凭证：**
- 求职者打开 PlanetX → "查看我的信任档案" → 看到自己的行为统计、声明状态、人格雷达图
- HR 打开 T-space → 候选人详情 → 看到行为证据链（而非仅简历）
- 第三方通过 MCP/API → 用 share_code 查询候选人的可验证声明

### 1.6 权威数据源：信任验证的公共基础设施

中国档案电子化行业存在一个产业断层：

| 档案类型 | 采集方 | 数据规模 | 当前状态 |
|----------|--------|:---:|------|
| 学籍档案 | 各省教育厅、学信网 | 亿级 | 仅验证查询，无商业应用 |
| 技能培训档案 | 人社部备案机构、职培平台 | 千万级 | 证书验证，零变现能力 |
| 人事档案 | 各级人才市场、央企档案室 | 亿级 | 纸质转电子，纯成本中心 |
| 继续教育/职业资格 | 各行业继续教育平台 | 千万级 | 满足合规要求，无商业出口 |

**这些机构的共同困境**：数据电子化了，但没有应用场景——他们不知道对接谁能变现。本质上是一个"数据采集完整、推向市场无能"的产业断层。

**looma 恰好填补了这个断层。** 档案数据进入 looma 后，不是作为"一份 PDF 附件"挂靠在简历上，而是作为**一条可验证的信任声明**被 Trust Agent 引用：

```
学信网学历数据 ──→  verification_source="chsi_credential"
                 ──→  claim_type="education", evidence_type="credential"
                 ──→  verification_status="verified_by_authority"

人社部技能证书 ──→  verification_source="mohrss_credential"
                 ──→  claim_type="skill", evidence_type="credential"
                 ──→  verification_status="verified_by_authority"
```

**证据类型扩展**：Trust Agent 的 `evidence_type` 字段新增 `"credential"` 类型——区别于 quiz（行为分析）、fleet_consensus（多方共识）、dialogue_analysis（沟通分析）等内生证据，credential 代表来自权威第三方的外部背书。

**为什么档案机构愿意接入 looma 而非招聘平台：**

| 对接 Boss 直聘 | 对接 looma |
|---|---|
| "把你的数据给我们填充简历" | "你的数据作为信任验证源被引用" |
| 数据所有权转移，一次性交易 | 数据所有权保留，按验证量持续分润 |
| 沦为简历数据供应商 | 成为信任基础设施的一部分 |
| 被平台替代风险 | 与平台共生 |

**按验证量分润的商业模型：**
- 求职者授权 looma 调取档案数据（一次 consent，可撤回）
- looma 向档案机构支付验证费（¥0.5-2/次）
- Trust Agent 将权威数据与求职者行为数据交叉验证，生成 verified 级别声明
- 雇主查看候选人信任档案时，完整验证详情为增值服务
- looma 收入中提取分润给档案机构

**三层信任模型中的权威数据定位：**

| 层 | 原来的数据 | 加入权威数据后 |
|----|-----------|---------------|
| **几何层** | 社交可达性 | 不变 |
| **记忆层** | quiz/fleet/match/resume/ask/share 行为快照 | + credential 验证记录 |
| **凭证层** | Trust Agent 规则引擎声明 | + `verified_by_authority` 级别声明，置信度高一个数量级 |

权威数据不是替代行为数据，而是为行为数据提供**外部锚点**——一个有学信网背书的学历声明 + 舰队协作行为 = 比单独任一来源都强的信任信号。

## 二、我们的核心命题

### 2.1 一句话命题

> **人与人的相处建立在社会化交集里的短暂记忆上。我们要把这些短暂交集长期记忆体化，形成可审计、可回放、可被智能体检验的信任验证凭证。**

### 2.2 信任的重新定义

```
今天的信任 = 你是谁（出身） + 你认识谁（人脉）
我们的信任 = 你做过什么（行为） + 这些行为是否自洽（验证）
```

这不是"给所有人打分"——而是让每个人都能用自己的行为记录来支撑自己的职业主张。雇主看到的不是我们打的分数，而是被多方共同产生、不可伪造的行为证据。

### 2.3 信任的三层模型

| 层 | 名称 | 强度 | 回答的问题 |
|----|------|------|------------|
| **几何层** | 社交可达性 | 弱 | A 与 B 结构上是否曾通过某条人际链相连？ |
| **记忆层** | 信任记忆体 | 强 | 交集当时发生了什么（文字/对话/行为）？ |
| **凭证层** | 可验证声明 | 强 | 智能体能否用记忆体支撑某个 identity claim？ |

- 几何层只是**关系上下文**，不是信任本身
- 用户可见的"信用"必须来自**记忆层 + 凭证层**
- 仅有"1 度人脉"不能自动验证任何 claim

## 三、我们选择的场景

### 3.1 为什么从职业成长切入

职业成长是"行为信任"最自然的试验场：

- **需求刚性**：求职是每个人都会经历的、需要"被验证"的场景
- **行为密集**：求职过程中自然产生大量互动（测评、组队、匹配、对话、共识确认）
- **验证有价**：雇主愿意为"可验证的行为画像"付费——信任在这里有商业价值
- **社会意义**：职业公平是最能触动个体和社会神经的信任议题

### 3.2 双品牌场景矩阵

| 品牌 | 面向 | 场景 | 产生什么记忆体 |
|------|------|------|----------------|
| **PlanetX** | C端求职者 | 人格测评闯关 | `personality_completion` — 你是谁 |
| | | 舰队组队闯关 | `fleet_co_presence` — 你和谁共事过 |
| | | Match 匹配扫描 | `match_scan` + `consensus_exchange` — 你被谁验证过 |
| | | Ask 对话 | `dialogue_session` — 你如何沟通与承诺 |
| | | 分享传播 | `share_signal` / `referral_binding` — 你如何影响他人 |
| **T-space** | B端 HR / 合伙人 | 企业入驻与管理 | 组织信用画像 |
| | | 候选人管理 | `partner_candidate_contact` — 谁关注过你 |
| | | AI 简历匹配 | 匹配行为记录 |
| | | 合伙人撮合 | 撮合接触记忆 |

### 3.3 为什么选择游戏化闯关而不是纯问答

- 游戏化行为**难以伪装**：做 30 分钟的人格闯关 vs. 填 5 分钟问卷，产生的是不同质量的行为数据
- 舰队协作**产生多方验证**：你的协作模式不是你自己声称的，是队友共同见证的
- 共识确认**是信任的最小原子**：两个人同时确认"我们合作愉快"，这是比任何评分都强的信任信号

## 四、我们的技术路线

### 4.1 技术选型原则

| 原则 | 含义 |
|------|------|
| **信任本体优先** | 技术服务于信任命题，不做为技术而技术的炫技 |
| **契约先行** | API 设计先写 JSON Schema，再写代码——接口是团队协作的宪法 |
| **渐进复杂度** | MVP 用 SQLite 够用就绝不上 PostgreSQL；能嵌入就不分布式 |
| **可验证性** | 记忆体 append-only、不可篡改；代码可审计；信任逻辑可解释 |
| **双端平等** | 大陆版和海外版在核心体验（LLM、QCC、信任逻辑）上保持等价 |

## 五、我们的不做什么

### 5.1 技术上的不做

| 不做 | 原因 |
|------|------|
| 不做用户可见的 trust_score | 打分是黑箱，违背可解释性——我们输出的是"是否支撑 claim"，不是数字 |
| 不做好友推荐 / 人脉扩展 | 图扩张是手段不是目的，信任不需要靠加好友来证明 |
| 不做 AI 生成简历 | 美化 PDF 不是验证能力——这是竞品的路 |
| 不做过早的分布式架构 | SQLite 够用就绝不分库分表——复杂度是债务不是资产 |

### 5.2 产品上的不做

| 不做 | 原因 |
|------|------|
| 不做内推计数 / 活跃度排行 | 行为的量不等于行为的质 |
| 不做会员等级绑定信任分 | 付费不能购买信任 |
| 不做纯静态简历解析 | 简历是入口，不是信任的真源 |
| 不做招聘信息流广告 | 商业模式不能污染信任信号 |

### 5.3 数据上的不做

| 不做 | 原因 |
|------|------|
| 不爬取竞品公开数据训练模型 | 训练数据来自已标注的记忆体 |
| 不使用用户行为数据做广告推荐 | consent 明确限定用途 |
| 不将信任数据出售给第三方 | 信任数据是用户的资产，不是我们的商品 |
