## 一、项目背景

### 1.1 从理论到工程

六度分隔理论（Six Degrees of Separation）指出：地球上任何两个陌生人之间，平均只需通过 5 个中间人（6 次连接）即可建立联系。这一理论经历了三个阶段的演进：

| 阶段 | 时间 | 贡献者 | 方法 | 结论 |
|------|------|--------|------|------|
| 实证 | 1967 | Stanley Milgram | 邮件链实验（奥马哈到波士顿） | 平均 5.2 个中间人 |
| 建模 | 1998 | Watts & Strogatz | WS 小世界模型（图论证明） | 高聚类 + 短路径的拓扑结构 |
| 大数据验证 | 2011 | Facebook 研究团队 | HyperANF 算法（7.21 亿用户） | 平均 **4.74 度**分隔 |

在社交媒体时代，这个数字进一步缩小。核心数学原理是：**网络规模增长时，平均路径长度以对数速度增加**——这是无标度网络（Barabási-Albert 模型）的"富者愈富"效应所致。

### 1.2 为什么 looma-zervi 需要它

looma-zervi 已经天然具备了社交网络的数据基础：

- **推荐关系**（`invite_codes`）：用户通过邀请码注册，形成有向推荐链
- **舰队共属**（`fleet_members`）：同一舰队的成员形成紧密社群
- **企业共属**（`enterprise_users`）：同一企业的用户形成组织关系

这三层数据构成了一个真实的社交图谱。六度分隔算法的价值在于：

1. **信任传递量化** — 推荐链越短，信任度越高，可直接用于推荐权重计算
2. **社交网络健康度监控** — 内测期间追踪平均路径长度、可达比例等指标
3. **Hub 节点发现** — 识别"超级连接者"，这些用户是增长的关键节点
4. **冷启动优化** — 新用户注册后，可立即计算其与网络中其他用户的连接关系

## 二、改造架构

### 2.1 设计原则

```
零侵入 → 不新建表、不改现有代码
复用认证 → 所有 API 走现有 @require_auth JWT 中间件
纯读取 → 从现有三张表读取数据构建图谱，不写入
可缓存 → 预留 TTL 缓存接口，数据量增长后可平滑扩展
```

```

│                    现有数据库                         │
│                                                     │
│  invite_codes     fleet_members    enterprise_users  │
│  (推荐关系)        (舰队共属)        (企业共属)        │
│      │                │                  │           │
│      ▼                ▼                  ▼           │
│  ┌─────────────────────────────────────────────┐    │
│  │        graph_builder.py                     │    │
│  │   构建无向图邻接表 {user: {neighbors}}        │    │

│                     │                               │
│  ┌──────────────────▼──────────────────────────┐    │
│  │         social_bfs.py                        │    │
│  │  BFS 最短路径 / 信任度评分 / 可达性统计       │    │
│  │  网络拓扑分析（聚类系数、Hub 排名）           │    │

│  ┌──────────────────▼──────────────────────────┐    │

│  │  4 个 REST API 端点（/v1/social/*）          │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

### 2.3 边的构建规则

| 数据源 | 表 | 边类型 | 构建逻辑 |
|--------|-----|--------|---------|
| 推荐关系 | `invite_codes` | 有向→无向 | `created_by ↔ used_by` 双向连接 |
| 舰队共属 | `fleet_members` | 团聚边 | 同一 `fleet_id` 的所有成员两两互连 |
| 企业共属 | `enterprise_users` | 团聚边 | 同一 `enterprise_id` 的所有成员两两互连 |

> **注意**：舰队和企业成员互连是 O(n²) 操作，适用于内测阶段（每个组织 < 100 人）。大规模时需改为星型连接（通过 captain/admin 中转）。

## 四、API 端点说明

所有端点均需 JWT 认证（`Authorization: Bearer <token>`）。

### 4.1 查找最短推荐链

```
GET /v1/social/connection/<target_user_id>?max_depth=6
```

**用途**：查找当前登录用户到目标用户的最短社交路径，返回详细推荐链。

**响应示例**：
```json
{
  "connected": true,
  "degrees": 2,
  "path": ["user_a", "user_b", "user_c"],
  "chain": [
    {"step": 0, "user_id": "user_a", "display_name": "张三", "role": "起点"},
    {"step": 1, "user_id": "user_b", "display_name": "李四", "role": "中间人 1"},
    {"step": 2, "user_id": "user_c", "display_name": "王五", "role": "终点"}
  ],
  "trust_score": 70,
  "message": "通过 2 个中间人建立连接"
}
```

### 4.2 计算分隔度数

```
GET /v1/social/degrees/<target_user_id>
```

**用途**：快速计算分隔度数和信任度评分（不返回完整路径，比 connection 更轻量）。

### 4.3 可达用户统计

```
GET /v1/social/reachable?max_depth=6
```

**用途**：统计当前用户在 N 步内可达的所有用户，按度数分组。

**响应示例**：
```json

    "1": 2,
```

## 五、信任度评分体系

### 5.1 评分规则

信任度评分基于社交距离，距离越近信任度越高：

```
分隔度数    信任度    含义
─────────────────────────────────────
  0 度       100     自己（完全信任）
  1 度        90     直接推荐（你邀请的人 / 邀请你的人）
  2 度        70     朋友的朋友
  3 度        50     朋友的推荐人
  4 度        30     弱关系
  5 度        15     远程连接
  6 度         5     六度分隔极限
  >6 度        0     超出六度，不可达
```

### 5.2 评分公式

```python
trust_score = max(0, 100 - (degrees - 1) * 20)  # 线性衰减
# 特殊处理：0 度 = 100（自己），1 度 = 90（直接推荐给予额外信任）
```

实际实现为查表法（`compute_trust_score` 函数），避免浮点误差。

### 5.3 应用场景

| 场景 | 信任度阈值 | 说明 |
|------|-----------|------|
| 推荐权重计算 | ≥ 50（≤3度） | 3 度以内的推荐人才计入推荐权重 |
| 企业背书传递 | ≥ 70（≤2度） | 2 度以内的信任才传递企业品牌背书 |
| 反欺诈检查 | ≥ 90（≤1度） | 只有直接推荐关系才豁免反欺诈检查 |
| 社交推荐展示 | ≥ 15（≤5度） | 5 度以内的连接可在"你可能认识的人"中展示 |

### 6.5 关键预警场景

#### 场景一：平均路径长度突然增大

**现象**：`avg_path_length` 从 2.0 跳到 5.0+

**原因分析**：
- 新用户注册但没有通过邀请码（成为孤立节点）
- 舰队/企业成员关系未正确建立

**处理**：
1. 检查 `invite_codes` 表是否有未关联的 `used_by`
2. 检查新注册用户的 `fleet_id` / `enterprise_id` 是否已分配
3. 考虑在前端引导新用户完成"加入舰队"步骤

#### 场景二：Hub 度数集中度过高

**现象**：Top-1 Hub 的度数占总边数 > 50%

**原因分析**：
- 某个用户（如创始人）邀请了绝大多数人
- 所有用户都加入了同一个舰队

**处理**：
1. 鼓励多个邀请入口（不同团队成员各自发邀请码）
2. 拆分过大舰队为多个小舰队
3. 这不是错误，但意味着该 Hub 用户离场会导致网络碎片化

#### 场景三：孤立节点增多

**现象**：`isolated` > 总用户数 10%

**原因分析**：
- 小程序直接注册的用户没有邀请关系
- 企业/舰队未分配

**处理**：
1. 为无邀请码的用户自动分配"系统邀请码"
2. 新用户注册后强制选择或创建舰队
3. 在 `graph_builder.py` 中考虑加入"同注册日期"的弱连接边

## 八、算法原理速查

### 8.1 BFS 广度优先搜索

核心路径查找算法，时间复杂度 O(V+E)：

```
1. 从起点出发，加入队列
2. 弹出队首节点，检查其所有邻居
3. 如果邻居是目标 → 返回路径
4. 如果邻居未访问 → 标记已访问，加入队列
5. 重复直到找到目标或队列为空
```

### 8.2 信任度衰减模型

```
信任度 = f(社交距离)

  100 ─────
   90 ─────
   70 ─────
   50 ─────
   30 ─────
   15 ─────
    5 ─────
    0 ──────────────────
    0  1  2  3  4  5  6  >6
         分隔度数
```

这是离散查表模型。后续可升级为连续衰减函数：

```python
# 可选的连续衰减模型（Phase 2 考虑）
trust = 100 * math.exp(-0.8 * degrees)
# 0度=100, 1度=44.9, 2度=20.2, 3度=9.1, 4度=4.1, 5度=1.8, 6度=0.8
```

### 8.3 网络拓扑指标

| 指标 | 公式 | 含义 |
|------|------|------|
| 平均度数 | `2E / N` | 每个用户平均有多少连接 |
| 平均路径长度 | `Σ dist(i,j) / (N*(N-1)/2)` | 任意两用户间的平均最短距离 |
| 6步可达比例 | `reachable_pairs / total_pairs * 100%` | 网络连通性指标 |
| Hub 度数集中度 | `max_degree / total_edges` | 网络对单点依赖程度 |

## 十、常见问题

### Q1: 新用户注册后为什么在社交图谱中不可见？

**A**: 新用户需要有至少一条社交边才会出现在图谱中。确保以下任一条件满足：
- 通过邀请码注册（`invite_codes.used_by` 有值）
- 加入了一个舰队（`fleet_members` 有记录）
- 加入了一个企业（`enterprise_users` 有记录）

### Q2: 信任度评分可以自定义吗？

**A**: 可以。修改 `social_bfs.py` 中的 `compute_trust_score` 函数的 `trust_map` 字典即可。建议在内测稳定后再调整，避免频繁变动影响数据可比性。

### Q3: 数据量大了之后性能怎么样？

**A**: BFS 的时间复杂度是 O(V+E)，1000 用户级别毫秒级完成。`_get_graph()` 每次请求重建图谱，1000 用户约 10ms。超过 10000 用户后需要加缓存（Phase 2）。

### Q4: 如何在管理后台展示社交网络可视化？

**A**: 调用 `/v1/social/network-stats` 获取邻接数据，前端用 D3.js force-directed graph 渲染。Phase 3 会提供前端组件。

### Q5: 这个算法和 DemoPPI 的治理权重有关系吗？

**A**: 有协同空间。DemoPPI 的治理权重 = 声誉×60% + 贡献×40%，而社交信任度可以作为"声誉"的一个因子。后续可跨项目打通。
