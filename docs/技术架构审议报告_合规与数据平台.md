## 一、审议摘要

### 1.1 议题背景

PlanetX / T-space 产品生态正从「内测闭环验证」向「多能力平台」演进，涉及：

1. **简历生成与优化 AI**（市场已成熟，差异化不在通用 LLM）
2. **多格式文档解析**（Markdown、PDF/DOCX、扫描件、复杂版式简历与 JD）
3. **征信与多维数据比对**（司法、社保、职业评价等第三方数据源接入）
4. **个人信息保护合规**（《个人信息保护法》及配套规范持续收紧）
5. **可选的区块链能力**（审计锚定、凭证验真，非平台前提）

同时需明确：**是否引入 LangChain、是否采用 MCP（FastMCP）、区块链在架构中的位置**。

### 1.2 审议结论（建议稿）

| 层级 | 建议 | 优先级 |
|------|------|--------|
| **L1 合规底座** | 授权（Consent）、数据分级、审计留痕、脱敏/加密、留存策略 — **最先建设** | P0 |
| **L2 数据适配层** | `ProviderPort` 模式对接征信/司法/社保等 API，LLM 仅作解释层 | P1 |
| **L3 文档流水线** | 格式路由 → OCR/版式 → 归一化 → Schema 抽取 → 校验 | P1 |
| **L4 AI 编排** | **MCP（FastMCP）+ 规则引擎 + 轻量 LLM 调用**；不选 LangChain 为平台底座 | P1–P2 |
| **L5 区块链（可选）** | 仅审计哈希锚定 / W3C 可验证凭证；**不做架构前提** | P3（按需） |

**一句话**：主战场在合规与数据集成，不在 Agent 框架；LangChain 可局部试用，不宜作为平台底座；MCP 用于工具协议化；区块链窄场景可选。

### 1.3 不建议的方向

| 方向 | 理由 |
|------|------|
| LangChain 作为全平台 Orchestration 底座 | 与 PIPL 可审计要求冲突；与现有 rules-first 设计重复；依赖重、版本变动快 |
| 征信/司法数据靠 LLM「训练知识」评估 | 不可溯源、不可合规；当前 `credit_routes` 的 LLM 模式仅为 MVP 占位 |
| 简历/征信明文上公链 | 与最小必要、目的限定原则冲突；监管风险高 |
| SDK 内置 FastMCP 1.0（`mcp.server.fastmcp`） | 能力陈旧；应选 Standalone FastMCP 2.x/3.x |

## 二、现状评估

### 2.1 代码库 AI 依赖度（2026-07-01 基线）

```
┌─────────────────────────────────────────────────────────────┐
│  不依赖 LLM（内测主路径可独立运行）                           │
│  · 认证 / 邀请 / 支付 / 配额 / 埋点                          │
│  · 游戏化（XP / 任务 / 舰队）                                  │
│  · 小程序人格测试（前端规则计分 → profile-sync）               │
│  · 叙事引擎（domain_engine + narrative_content 手工文案）    │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────────┐
│  混合模式（规则优先，LLM 增强）                                │
│  · central_brain：关键词意图 → LLM 兜底                        │
│  · MBTI / 诗词 / 心理层：规则/检索为主                         │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────────┐
│  强依赖 LLM                                                    │
│  · /v1/ask RAG、Navigator 对话                                 │
│  · 简历/JD 解析与匹配、征信文本分析、简历优化                     │
└─────────────────────────────────────────────────────────────┘
```

**量化估算**（按 API 路由）：约 **70%** 可在无 LLM 下运行；约 **15%** 有规则降级；约 **15%** 强依赖 LLM。

### 2.2 与 MCP 协议的关系

- 当前 **未实现 MCP**；存在自研 `central_brain.dispatch()` → 内部 capability port 模式。
- 概念上类似 Agent 工具路由，但 **非 MCP 标准**（无 JSON-RPC、无 `tools/list`、单体进程内调用）。
- 建议：**Standalone FastMCP Sidecar**，与 Flask REST 并行，共享 `src/agents/*` 业务逻辑。

### 2.3 关键代码风险点

| 模块 | 文件 | 风险 |
|------|------|------|
| 文档解析 | `ingest/markitdown_convert.py` | 电子版 OK；扫描件/复杂表格需 OCR 层 |
| 征信 | `api/routes/credit_routes.py` | LLM 从文本/公司名「猜测」评估，无法对接真实数据源 |
| LLM 调用 | `agents/central_brain.py` | `requests` 直调 DeepSeek/OpenAI/Ollama，无统一 consent/audit 管道 |
| 依赖 | `requirements.txt` | 无 LangChain / LiteLLM（市场调研文档与代码不一致） |

## 三、分业务线审议意见

### 3.1 简历生成与优化 AI

| 维度 | 审议意见 |
|------|----------|
| 市场 | 红海，WPS/超级简历/各类 API 已成熟 |
| 差异化 | PlanetX **人格标签 + 叙事语境 + 合规授权下的个性化**，非通用模板 |
| 技术 | 结构化输出（JSON Schema / Pydantic）+ 评测集 + A/B；**不需要 LangChain** |
| 建议 | 简历优化作为 **L4 AI 编排** 的一个 MCP Tool，输入需经 L1 合规 Gate |

### 3.2 文档解析（多格式简历 / JD）

| 维度 | 审议意见 |
|------|----------|
| 性质 | **Pipeline 工程**，非 Agent 框架问题 |
| 推荐流水线 | 上传 → 格式路由 → OCR/版式 → 归一化 MD/JSON → Schema 抽取 → 置信度 + 人工复核 |
| 现有基础 | MarkItDown 覆盖 PDF/DOCX/XLSX 等电子版 |
| 缺口 | PaddleOCR / Azure Document Intelligence / Unstructured 等 OCR 与版式层 |
| 建议 | 独立 **`DocumentPipeline` 服务模块**（L3），Flask route 与 MCP tool 共用 |

### 3.3 征信 / 司法 / 社保 / 职业评价数据比对

| 维度 | 审议意见 |
|------|----------|
| 性质 | **B2B 数据集成 + 合规产品** |
| 核心组件 | Provider Adapter、Consent Gate、Match Engine、Audit Log、Data Minimization |
| LLM 角色 | **仅**生成差异说明、风险提示摘要；**不能**作为数据来源 |
| 演进路径 | `credit_routes` 从 LLM 占位 → **`CreditProviderPort` 接口 + 多 Adapter** |

**建议接口形态（草案）**：

```python
# 示意 — 供会议讨论，非最终实现
class CreditProviderPort(Protocol):
    provider_id: str          # e.g. "qichacha", "judicial_open"
    required_consent_scopes: list[str]

    def query(self, *, entity_id: str, scopes: list[str], consent_id: str) -> NormalizedReport: ...
    def health(self) -> ProviderHealth: ...
```

每次查询链路：`consent 校验 → adapter.call → normalize → rule_match → audit 写入 → LLM explain（可选）`

### 3.4 个人信息保护（PIPL）与技术手段

| 要求 | 技术手段建议 |
|------|-------------|
| 单独同意 / 目的限定 | Consent 表 + 场景码（`resume_parse` / `credit_check` / `share_profile`） |
| 最小必要 | 字段级授权；Adapter 只请求授权范围内的字段 |
| 可撤回 | Consent 状态机 + 下游数据 TTL 删除任务 |
| 审计可追溯 | 不可篡改审计日志（见 L5 可选哈希锚定） |
| LLM 调用前脱敏 | Prompt Redaction Middleware：姓名/身份证/手机号替换为 token |
| 日志与埋点 | `product_events` 不含 PII；properties 白名单 |

**原则**：合规层是 **横切关注点**，必须在所有 AI / 数据能力之上，不可事后补丁。

### 3.5 区块链（可选模块）

| 场景 | 建议 | 优先级 |
|------|------|--------|
| 审计不可篡改 | 链下加密存储 + **链上哈希锚定** | P3 可选 |
| 学历/证书验真 | W3C Verifiable Credentials + 权威机构签名 | P3 按需 |
| 授权事件存证 | Consent 事件签名 + 哈希锚定 | P3 可选 |
| 简历/征信明文上链 | **不建议** | — |
| Token 激励上传敏感数据 | **不建议** | — |

**结论**：区块链 **不作为平台架构前提**；若做，仅限「Hash Anchor + Off-chain Encrypted Vault」窄场景。

## 四、框架选型审议

### 4.1 LangChain — 建议：**不选为平台底座**

| 对比项 | LangChain | 本项目需求 |
|--------|-----------|------------|
| 合规可审计 | 弱（Agent 黑盒、prompt 易漏 PII） | 强 |
| 确定性 | Agent 行为难预测 | 征信比对需 deterministic |
| 与 MCP | 功能重叠（tool calling） | 已倾向 FastMCP |
| 中文 OCR/文档 | 非核心能力 | 核心痛点 |
| 维护成本 | 依赖重、API 变动快 | Flask 单体求稳 |

**可局部使用的替代**：

| 场景 | 建议方案 |
|------|----------|
| 结构化 LLM 输出 | Instructor / Pydantic AI / JSON Schema |
| 有状态多步工作流 | 现有 `domain_engine` + `act1_state_machine`；或 LangGraph **仅 Navigator 试点** |
| RAG | ChromaDB + 自研 retrieval + reranker |
| 长任务编排 | Temporal / Celery（重试、审计、超时） |

### 4.2 MCP — 建议：**Standalone FastMCP Sidecar**

| 选项 | 建议 |
|------|------|
| Standalone FastMCP 2.x/3.x | ✅ 首选 — 快速包装 capability port，支持 stdio / Streamable HTTP |
| 官方 MCP Python SDK 直接使用 | 仅当需协议级定制或 SDK 2.0 稳定后再评估 |
| SDK 内置 FastMCP 1.0 | ❌ 不选 |

**与 Flask 共存架构**：

```
小程序 / SaaS ──REST──► Flask（现有，C/B 端主入口）
Cursor / Agent ──MCP──► FastMCP Sidecar（新增）
                              │
                              └── 共享 src/agents/* + 未来 Compliance Gate
```

**首批 MCP Tool 建议（P1）**：

| Tool | 对应现有模块 | 是否需 Consent |
|------|-------------|----------------|
| `rag_query` | `search_chroma` + LLM | 视知识库内容 |
| `match_jobs` | `job_match_pipeline` | 简历数据 |
| `parse_resume` | `document_agents` | 简历上传 |
| `search_poems` | `poetry_search` | 低 |
| `analyze_mbti` | `mbti_analyzer` | 用户文本 |

**不宜 MCP 化**：game CRUD、payment、analytics 等确定性 API。

### 4.3 轻量 LLM 调用 — 建议：**保留并增强，不替换为 LangChain**

- 保留 `central_brain._call_llm()` 多 Provider fallback 模式。
- 增强点：调用前 **Consent 校验 + PII Redaction**；调用后 **Audit 写入**；结构化输出统一走 Schema 校验。

## 五、目标架构（审议建议稿）

```
┌──────────────────────────────────────────────────────────────────┐
│  C 端: 小程序 / PlanetX Web    B 端: T-space SaaS    Agent 客户端   │
└────────────┬─────────────────────────┬─────────────────────────────┘
             │ REST                    │ MCP (FastMCP)
┌────────────▼─────────────────────────▼─────────────────────────────┐
│                    API Gateway + Compliance Gate (L1)                 │
│   auth · consent · PII redaction · audit · rate limit · retention    │
└────────────┬─────────────────────────────────────────────────────────┘
             │
    ┌────────┼────────────┬─────────────────┬──────────────────┐
    ▼        ▼            ▼                 ▼                  ▼
 Document  Credit/Data   Game/Rules      RAG/LLM           Audit Store
 Pipeline  Adapters (L2) Engine (L4)     Service (L4)      (+ hash anchor L5)
   (L3)
```

### 5.1 建设阶段建议

| 阶段 | 时间框 | 交付物 | 依赖 |
|------|--------|--------|------|
| **P0** | 0–3 月 | Compliance Gate MVP（consent 表、audit 表、prompt 脱敏中间件） | 无 |
| **P0** | 0–3 月 | FastMCP Sidecar + 3 个 tool（rag / match_jobs / parse_resume） | 业务逻辑抽离 `current_app` |
| **P1** | 3–9 月 | `DocumentPipeline` v1（MarkItDown + OCR 插件位 + Schema 校验） | P0 |
| **P1** | 3–9 月 | `CreditProviderPort` 接口 + 1 个真实 Adapter PoC | 资质 / 商务 |
| **P2** | 9–18 月 | Navigator → MCP Host loop（规则引擎 + MCP tools） | P0/P1 |
| **P2** | 9–18 月 | 简历优化差异化（人格 + 叙事注入）+ 解析 benchmark | P1 |
| **P3** | 按需 | 审计哈希锚定 / VC 验真 | 法务确认 |

### 5.2 P0 Compliance Gate 最小范围（建议）

| 组件 | 说明 |
|------|------|
| `consents` 表 | user_id, scope, purpose, granted_at, revoked_at, ip, user_agent |
| `audit_logs` 表 | actor, action, resource_type, resource_id, consent_id, metadata（无 PII） |
| `@require_consent(scope)` 装饰器 | 挂于 resume/credit/ask 等敏感 route |
| `redact_pii(text)` | LLM 调用前替换身份证/手机/邮箱 |
| 留存策略 | 敏感 raw 文本 TTL（如 30/90 天可配置） |

## 六、风险与依赖

| 风险 | 等级 | 缓解 |
|------|------|------|
| 征信数据源资质与成本 | 高 | 先接口抽象 + 1 家 PoC；LLM 路径标注「非正式数据源」 |
| OCR 中文简历准确率 | 中 | 标注数据集 + 人工复核队列 |

| LangChain 诱惑（「别人都在用」） | 低 | 本文审议约束；局部试点需 ADR |
